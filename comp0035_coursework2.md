# Coursework 2
## Requirements definition and analysis
### Requirements identification methods
From the 9 techniques listed in BAKOK, the following techniques will be used in the project. (BABOK, 2015) 

1. **Brainstorming**: The group will have a discussion and share ideas to derive creative ideas for the problem. It is applied to the project because the project group has 5 people from diverse background and they are willing and engaged in the participation.
2. **Interview**: Through conversations with potential users, capture the requirements from question and answer. It helps the team to discover some hidden requirements like non-verbal aspects in a simple and direct way. Since the project group is full of experienced product manager and developers, no prior training needed for interview.
3. **Prototyping**: Throughout the life cycle of the product development, capture requirements from each iterations. It can perfectly incorporate with Scrum. For each sprint, the requirements will be evaluated with user stories and validate it in the next sprint. It has smaller costs because it can have early feedback and aviod infeasible solutions. Since the project group is full of experienced product manager and developers, they are equiped with technology knowledge and able to solve complex case through iterations.
4. **Requirements Workshops**: Elicit requirements from creative ideas generated in a structured session with users and stakeholders. The workshop can be finished in a short period of time and costs less than interviews. This will perfectly incorporate with Scrum and finish the sprint on time. In addition, because the project group is full of experienced product manager and developers, they are able to conduct the workshop successfully.
5. **Survey/Questionnaire**: Collect information about the products and requirements after many users or stakeholders finish the written questions in a short period of time. This will perfectly incorporate with Scrum and finish the sprint on time. Closed and open questions can provide quantitative and qualitative insights.

However, the following techniques cannot be used in the project.
1. **Document Analysis**: Analyse some documents like market studies and existing solutions to have some ideas about the product. However, it cannot be applied because currently there is no any volcano apps in the market which has similar features. In addition, it's hard to find any market studies because the number of target users and market are quite small. Due to the nature of Scrum, each sprint should be finished in 1-4 weeks. There is no enough time for the group to work on the time-consuming analysis.
2. **Focus Group**: Through the interactions with a group of users, capture their needs and perferences. However, the interactive session costs a lot in sceduling the session, especially during the pandemic. Since the volcano app is a new product and there is no existing solutions in the market, therefore focus group is not useful to identify the requirements. This technique can be considered after a few sprint for a more completed volcano app.
3. **Interface Analysis**: It used for the analysis on the interface interactions between solutions and components of solutions. However, the volcano app is a new product and there is no existing solutions in the market, therefore interface analysis is not applicable for now but could be useful in the future sprints.
4. **Observation**: Capture the requirements when the users are using existing process. However, this technique is time-consuming and can only improve the current process.

### Requirement specification method
User story is used to specify the requirements beacasue the project methodlogy selected is a combination of Scrum and CRISP-DM. User story is an important part in Agile method. (Abdollazadeh, 2021)
1. For each sprint in the Scrum, stories in the sprint help the team to focus on the real needs of the end users and have a better understanding.
2. The teams can work together and come out with creative solutions

### Prioritisation method
**100 points method**: Considering the project methodology selected is a combination of Scrum and CRISP-DM. Scrum requires each sprint to be finished in very short period of time, normally 1-4 weeks. It is essential to save time and iterate from the feedback. 100 points method is selected to measure the priority for different requirements with quantitative values in simple and direct way. In addition, the project group are 5 sensior developers and product managers, they will provide professional evaluation on the requirements. It is not necessary to include other prioritisation method because they can make good justification based on their own experience. They know what requirements are likely to be essential to the app aand they will give more points for higher priority.

### Major changes in target audience
After CW1, the target audience is changed because I have a clearer vision about the scope and use of the Volcano app. Now there are two different categories of the web users.
#### Category I: Researchers
Researchers such as geologists from government, companies or universities who are looking to access some historical volcanic eruption data to provide a base data trend for their prediction models to improve the accuracy of their predictions. More accurate predictions can save a lot of damage and protect the residents lived near volcano.

#### Category II: Residents lived near volcano
The app will send eruption alert according to the volcano prediction model to the residents. Meantime, the residents lived near volcano report the daily monitoring like temperature rise and ground vibrations for scientific research. In addition, the residents can also use the app to access the eruption data to simply know the trend of the volcano and the environment they lived with.

## Design
### Structure and flow of the interface
The interface will be designed as a desktop screen because it is easier for users to read the information and analyse data trend. The size will be 1920*1080 because it is currently the most popular screen resolution (23%) worldwide. (Desktop Screen Resolution Stats Worldwide, 2021)

The non-functional requirements (US01 & US02) will not be included in the interface design.

## Testing
### Choice of unit testing library
1. **Unittest**: Fast test collection and flexible test execution. 
2. **PyTest**: Special and simple class fixture for making testing easier.
3. **Doctest**: Python Interactive Shell for the command prompt and inclusive application.

PyTest's straightforward syntax makes it easy to run tests, and it's useful for a variety of purposes, including functional and API testing. It also has the ability to execute any subset of tests in parallel. (Real Python, 2021) Pytest provides a fixture concept that is both powerful and easy, unlike any other testing framework like Doctest. It can ensure specific environment for a single test (Knapsack, 2021)  Compared with unittest, both frameworks are excellent for testing in Python. However, pytest is more user-friendly. Pytest's code is simple, concise, and efficient. We'll need to import modules, build a class, then specify the testing functions within that class for unittest. In the case of pytest, however, we simply need to define the testing method. Pytest is likewise a quick and efficient programme. (Raj & Python Pool, 2021) Since in this project, the team decide to use Scrum, an agile methodology, a simple unit testing library can help the team save time for each sprint. Therefore, I will use **pytest** as the test library, and **pytest-cov** for coverage.
